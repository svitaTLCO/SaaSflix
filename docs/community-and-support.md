# Community and Support

## Objective

Create a product-native social and support layer that feels alive, useful, and tightly connected to the software ecosystem.

Community should increase trust, discovery, and belonging. Support should reduce friction and create reusable knowledge. Neither should feel bolted on.

## Product principle

Do not build a generic social network. Build a **software-native conversation system**.

Users should mainly interact around:
- products
- launches
- feedback
- support
- roadmap motion
- founder updates

## Core conversation types

Model conversations as structured thread types rather than completely unrelated systems.

Recommended thread types:

- release discussion
- product feedback
- support question
- roadmap discussion
- founder post discussion
- announcement thread
- known issue thread

This lets the UI and moderation logic adapt while sharing primitives.

## Founder feed

The founder feed is a signature product surface.

It should support:
- short updates
- longer notes
- linked releases
- linked app previews
- media
- scheduling
- visibility levels
- comments and reactions
- highlighted or pinned items

The feed should communicate momentum, not noise.

## Support model

Support should combine:
- private support where necessary
- public or semi-public answers where reusable
- known issues
- linked knowledge articles
- app-specific context
- lifecycle and status awareness

Avoid forcing every question into private tickets if a shared answer would help others.

## Product feedback model

Feedback should be structured enough to be useful:

- app association
- version context
- reproduction or use-case data where relevant
- visibility controls
- follow / watch
- status labels such as under review, planned, shipped, closed

## Moderation and health

The system should support:
- content reporting
- moderation queue
- soft hide / lock / archive
- abuse controls
- spam prevention
- author and staff role labeling
- transparency for moderation state where appropriate

## Knowledge base relationship

Support and community should feed the knowledge base.

Promote high-value solved content into:
- FAQ entries
- troubleshooting guides
- release caveats
- migration guides
- known issue summaries

## UX principles

### Clarity over chatter
Threads should be easy to scan and understand.

### Reuse over repetition
Answered support should become searchable and reusable.

### Context attachment
Whenever possible, community objects should attach to:
- an app,
- a release,
- a founder post,
- or an account/support context.

### Trust signals
Staff responses, accepted answers, known issue markers, and status changes should be visually obvious.

## Notification logic

Users may want updates on:
- followed apps
- watched threads
- replies or mentions
- support case changes
- new drops
- roadmap changes
- founder posts

Notification preferences should be configurable and avoid fatigue.

## Privacy and visibility

Threads may be:
- public
- member-only
- app-specific member-only
- staff/private support
- founder-circle limited

Visibility rules must be explicit.

## Metrics

Valuable metrics include:
- support response time
- accepted answer rate
- repeat issue frequency
- unresolved discussion age
- engagement by app or release
- knowledge article deflection impact

## Non-negotiable outcome

Community and support must make the platform feel more alive and more trustworthy, not more chaotic.
